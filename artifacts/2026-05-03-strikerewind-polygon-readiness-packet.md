## Direct answer to "ready for Polygon"

**No, not yet — but the gap is small and bounded.** Earlier handoffs described an `OptionsDataProvider` interface "ready behind a shim." That language was aspirational: there is no such interface in the repo today. What exists is a single warehouse column (`spread_results.data_source`) and a service that defaults that column to the literal string `'polygon'`. The actual options pricing path is fully synthetic and runs entirely on yfinance price history + Black-Scholes + a random-noise IV multiplier.

Three things have to land before flipping the data source switch produces honest output:

1. A real `OptionsDataProvider` interface (does not exist in code).
2. A `PolygonOptionsProvider` that implements it.
3. Replacement of `IVService.getIVData(...)` and `NewAnalysisEngine.estimateCreditSpreadPremium(...)` so they consume provider data instead of synthesizing it.

All three are mechanical once Travis greenlights. The packet below specifies each.

## Current state — built vs aspirational

### Built
- **Warehouse table** `spread_results` with `data_source TEXT` column (per `spreadResultsService.ts:46-47`). Today every row writes `data_source = 'polygon'` regardless of what produced it — the literal default in `analysisToWarehouseRows(...)`. This is misleading data lineage; ship date for the fix is post-Polygon-cutover.
- **Spread results service** (`spreadResultsService.ts`) — pure mapping function `analysisToWarehouseRows`, bulk upsert, hunt/matrix queries, batch-run lifecycle methods. Provider-agnostic by design. No change needed.
- **Scheduler cron pattern** (`schedulerService.ts:154-167`) — 9:30 AM ET market-open warmer + 6:00 AM ET backup, both Mon–Fri, `America/New_York` timezone. Iterates `SP500_TICKERS`, calls `analysisEngine.analyzeStock(ticker, 'both', 5, 52)` per ticker. The orchestration layer is reusable; only the inner per-ticker analysis call changes.
- **Tier flag plumbing** — `UsageServiceDB.tierLimits.{FREE,STANDARD,PRO}.hasIVData` exists end-to-end (frontend `UserContext` reads it; backend `index.ts:213` returns it). Wiring is fine; the data feeding it is the issue.

### Aspirational (not in code)
- `OptionsDataProvider` interface — does not exist anywhere under `backend/src/`. `Grep` for the symbol returns only matches in this packet's predecessors. Earlier handoffs described "a shim behind the interface" — there is no shim and no interface.
- "Polygon-ready abstraction layer" — `newAnalysisEngine.ts` calls `this.ivService.getIVData(ticker, prices)` directly. There is no abstraction to swap.
- "Provider-neutral premium estimation" — premium estimation is hard-coded Black-Scholes with a `marketDiscount = 0.7` constant (`newAnalysisEngine.ts:99`) and synthetic IV multipliers (`ivService.ts:59` `1.1 + (Math.random() * 0.4 - 0.2)`).

### Customer-facing risk created by the synthetic path

This is the credibility part. Three places where the current code makes a claim the data does not support:

1. **`STANDARD`/`PRO` tier advertises `hasIVData: true`** (`usageServiceDB.ts:38, 48`). Frontend gates a "current IV / IV rank / IV percentile" display on this flag. Today every paying user would see IV numbers generated by `Math.random() * 0.4 - 0.2`. That is not "real-time IV data" — it is HV with cosmetic noise. If the launch post or methodology block mentions IV, the words don't match the implementation.
2. **Methodology block currently reads "backtest of historical price data, not a forecast"** (landing page, shipped 2026-05-01). That phrasing was deliberately provider-neutral and is technically defensible — premiums are *estimated* from price history. But anyone who reads the code and sees `marketDiscount = 0.7` as a constant will not characterize the output as "backtest of options data."
3. **Hunt result payloads include `currentIV`, `ivRank`, `ivPercentile`** (`newAnalysisEngine.ts:467` and downstream). These ride through to the Hunt UI under the same paid-tier flag. Same problem as (1).

None of these are emergencies — Travis's marketing freeze means no traffic is hitting the synthetic path yet. But the gap is the reason "ready for Polygon" is the next product question, not just a checkbox.

### Commit 1 — `OptionsDataProvider` interface + fixture provider

New file `backend/src/services/providers/optionsDataProvider.ts`:

```typescript
export interface OptionChainQuote {
  ticker: string;
  asOfDate: string;       // YYYY-MM-DD
  expiration: string;     // YYYY-MM-DD
  strike: number;
  right: 'call' | 'put';
  bid: number;
  ask: number;
  mid: number;
  impliedVolatility: number; // annualized, decimal
  delta?: number;
  openInterest?: number;
}

export interface IVSnapshot {
  ticker: string;
  asOfDate: string;
  currentIV: number;       // annualized, percentage (0-100)
  ivRank: number;          // 0-100, percentile vs trailing 52w
  ivPercentile: number;    // 0-100, days-below over trailing 52w
}

export interface OptionsDataProvider {
  readonly name: string;   // 'polygon' | 'fixture' | future providers
  getIVSnapshot(ticker: string, asOf: Date): Promise<IVSnapshot>;
  getCreditSpreadQuote(args: {
    ticker: string;
    asOf: Date;
    expiration: Date;
    shortStrike: number;
    longStrike: number;
    right: 'call' | 'put';
  }): Promise<{ creditPerShare: number; impliedVolatility: number } | null>;
}
```

New file `backend/src/services/providers/fixtureOptionsProvider.ts` — wraps current Black-Scholes + synthetic-IV behavior behind the interface. Used in dev/test only. Lets `huntFixtureMode.test.ts` and `spreadResultsService.test.ts` keep running deterministically.

### Commit 2 — `PolygonOptionsProvider`

New file `backend/src/services/providers/polygonOptionsProvider.ts`. Single-file provider, ~150 lines. Responsibilities:

- Read `POLYGON_API_KEY` from env. Throw clearly on boot if missing in production.
- `getIVSnapshot(ticker, asOf)` → call Polygon options snapshot endpoint, derive IV rank/percentile from trailing 52w of daily IV (warehoused — see below).
- `getCreditSpreadQuote(...)` → call Polygon options chain endpoint for the asOf date, find the two contracts, return `(short.bid - long.ask)` per share. Returns `null` when liquidity is thin (no bid, OI < 10) — caller handles missing data without crashing.
- Internal rate limiter: token-bucket sized to whichever Polygon plan we land on (see plan question below). Logs but does not retry on 429 — the scheduler already handles per-ticker errors via its existing try/catch.
- New warehouse table `iv_snapshots(ticker, as_of_date, current_iv)` populated nightly, used by `getIVSnapshot` for rank/percentile math without re-fetching 252 historical days per ticker per request.

### Commit 3 — wire the provider through, retire synthetic path

Changes:
- `newAnalysisEngine.ts` constructor takes a `provider: OptionsDataProvider`. Replace `this.ivService.getIVData(...)` with `provider.getIVSnapshot(...)`. Replace `estimateCreditSpreadPremium(...)` body with a call to `provider.getCreditSpreadQuote(...)`; on `null` return, drop the cell from results (do not synthesize).
- `index.ts` boot: `const provider = process.env.NODE_ENV === 'production' ? new PolygonOptionsProvider() : new FixtureOptionsProvider()`. Pass into `NewAnalysisEngine`. (Tests already use fixture mode.)
- `schedulerService.ts` adds a nightly 4:30 PM ET task that snapshots IV for `SP500_TICKERS` into `iv_snapshots` so next morning's warming has trailing-52w data ready.
- `spreadResultsService.ts` `analysisToWarehouseRows` `dataSource` arg is set by the caller from `provider.name`, not defaulted — kills the misleading hard-coded `'polygon'` default.
- `IVService` deleted.
- `usageServiceDB.ts` `hasIVData` flag now matches reality (provider returns real IV, flag is honest).

## Load-bearing question — which Polygon plan

Earlier internal references cited "Polygon Starter, $29/mo." That's correct for stocks. **Options data on Polygon is a separate plan family.** AutoBox cannot finalize this without confirming current Polygon pricing and what each tier actually returns for *historical* options chains, since the product story is "this strike + DTE has been winning consistently over years." Specifically:

- A **current-snapshot-only** tier is enough to make `IVService` honest (today's IV, today's IV rank vs trailing nightly snapshots we collect ourselves). That alone closes the credibility gap on the `hasIVData` tier flag.
- A **historical-options-chains** tier is needed to make the *credit* number in `spread_results` reflect what someone could actually have collected on entry date X. Without it, premiums stay theoretical (Black-Scholes-derived); only IV becomes real. The methodology block can stay accurate either way, but "real backtest using historical option chain data" is only true with the higher tier.

**Recommendation:** start with whichever Polygon Options tier provides current options snapshots + at least 2 years of end-of-day historical chains, even if it's the higher Options tier. Reason: the synthetic-IV credibility gap is fixable cheaply, but the synthetic-*premium* gap is the one that breaks the product's core claim. Spending $29 to fix only IV and leaving premiums theoretical defeats the purpose of switching providers.

If current-only is the only affordable option, ship it (closes the IV gap, keeps premiums explicitly labeled "theoretical estimate" in copy and API responses) and queue the historical-chain upgrade as a Phase 2.

This question is the only thing AutoBox cannot answer alone before shipping. Travis confirms the plan when greenlighting Polygon post-`TESTING COMPLETE`.
