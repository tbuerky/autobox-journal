StrikeRewind — Polygon readiness packet

Date: 2026-05-03

---

Direct answer to "ready for Polygon"

No, not yet — but the gap is small and bounded. Earlier handoffs described an OptionsDataProvider interface "ready behind a shim." That language was aspirational: there is no such interface in the repo today. What exists is a single warehouse column (spread_results.data_source) and a service that defaults that column to the literal string 'polygon'. The actual options pricing path is fully synthetic and runs entirely on yfinance price history + Black-Scholes + a random-noise IV multiplier.

Three things have to land before flipping the data source switch produces honest output:

1. A real OptionsDataProvider interface (does not exist in code).

2. A PolygonOptionsProvider that implements it.

3. Replacement of IVService.getIVData(...) and NewAnalysisEngine.estimateCreditSpreadPremium(...) so they consume provider data instead of synthesizing it.

All three are mechanical once Travis greenlights. The packet below specifies each.

---

Current state — built vs aspirational

Built

  • Warehouse table spread_results with data_source TEXT column (per spreadResultsService.ts:46-47). Today every row writes data_source = 'polygon' regardless of what produced it — the literal default in analysisToWarehouseRows(...). This is misleading data lineage; ship date for the fix is post-Polygon-cutover.
  • Spread results service (spreadResultsService.ts) — pure mapping function analysisToWarehouseRows, bulk upsert, hunt/matrix queries, batch-run lifecycle methods. Provider-agnostic by design. No change needed.
  • Scheduler cron pattern (schedulerService.ts:154-167) — 9:30 AM ET market-open warmer + 6:00 AM ET backup, both Mon–Fri, America/New_York timezone. Iterates SP500_TICKERS, calls analysisEngine.analyzeStock(ticker, 'both', 5, 52) per ticker. The orchestration layer is reusable; only the inner per-ticker analysis call changes.
  • Tier flag plumbing — UsageServiceDB.tierLimits.{FREE,STANDARD,PRO}.hasIVData exists end-to-end (frontend UserContext reads it; backend index.ts:213 returns it). Wiring is fine; the data feeding it is the issue.

Aspirational (not in code)

  • OptionsDataProvider interface — does not exist anywhere under backend/src/. Grep for the symbol returns only matches in this packet's predecessors. Earlier handoffs described "a shim behind the interface" — there is no shim and no interface.
  • "Polygon-ready abstraction layer" — newAnalysisEngine.ts calls this.ivService.getIVData(ticker, prices) directly. There is no abstraction to swap.
  • "Provider-neutral premium estimation" — premium estimation is hard-coded Black-Scholes with a marketDiscount = 0.7 constant (newAnalysisEngine.ts:99) and synthetic IV multipliers (ivService.ts:59 1.1 + (Math.random() * 0.4 - 0.2)).

Customer-facing risk created by the synthetic path

This is the credibility part. Three places where the current code makes a claim the data does not support:

1. STANDARD/PRO tier advertises hasIVData: true (usageServiceDB.ts:38, 48). Frontend gates a "current IV / IV rank / IV percentile" display on this flag. Today every paying user would see IV numbers generated by Math.random() * 0.4 - 0.2. That is not "real-time IV data" — it is HV with cosmetic noise. If the launch post or methodology block mentions IV, the words don't match the implementation.

2. Methodology block currently reads "backtest of historical price data, not a forecast" (landing page, shipped 2026-05-01). That phrasing was deliberately provider-neutral and is technically defensible — premiums are estimated from price history. But anyone who reads the code and sees marketDiscount = 0.7 as a constant will not characterize the output as "backtest of options data."

3. Hunt result payloads include currentIV, ivRank, ivPercentile (newAnalysisEngine.ts:467 and downstream). These ride through to the Hunt UI under the same paid-tier flag. Same problem as (1).

None of these are emergencies — Travis's marketing freeze means no traffic is hitting the synthetic path yet. But the gap is the reason "ready for Polygon" is the next product question, not just a checkbox.
